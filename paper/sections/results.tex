This section reports the current prototype outputs. The results support a strengthened research direction, but not yet a claim of superiority over reproduced external systems.

\subsection{Trace Reconstruction}
The Toronto OSM trace probe currently contains $25{,}000$ parsed trackpoints with timestamp coverage $1.00$. The pipeline found $11$ pseudo-segments and $11$ useful segments, for a useful segment rate of $1.00$. Approximate map matching succeeded for all attempted useful segments. The median GPS-to-route median distance is approximately $6.99$ meters, indicating close geometric alignment between sampled points and reconstructed routes. However, the median route-distance ratio is approximately $2.37$, above the strict target ceiling of $2.0$.

Therefore, the current strict prototype usability flag is false. The exploratory usability flag is true under the looser $2.5$ route-distance-ratio ceiling. This distinction is central: the data is useful for exploratory pseudo-history experiments, but the reconstruction is not yet clean enough for strong claims about route fidelity.

\subsection{Path-Similarity Metrics}
Against held-out reconstructed route coordinates, the profile method achieves mean path F1 of $0.630$ and mean NDTW of $0.351$, compared with $0.601$ and $0.344$ for shortest-distance. The improvement is directionally promising but not yet publication-grade without a larger public benchmark and confidence intervals.

\subsection{Porto Public-Dataset Prototype}
To begin moving toward a same-data public benchmark, we also ran a Porto taxi trajectory prototype using the public ECML/PKDD taxi trajectory data. This experiment samples trips from the Porto training file, reconstructs an observed OSM driving route using simplified GPS anchors, generates up to eight diverse OSM driving-route candidates for the same origin and destination, and ranks the candidates using random, shortest-distance, generic profile, and vehicle-aware profile baselines. We also report two oracle upper bounds: a feature oracle, which ranks candidates by the same feature-distance criterion used to define the ranking oracle, and a path oracle, which ranks candidates by path overlap with the reconstructed route. These oracle rows diagnose candidate-pool quality and are not deployable methods. Prompt/SBERT and hybrid methods are skipped because Porto trajectories do not include natural-language route preferences.

The simplified-anchor reconstruction reduces mean observed route-distance ratio from the prior dense-stitching value of about $2.82$ to $2.69$ and the median from $2.17$ to $2.04$, while reducing the mean number of reconstruction anchors to 62\% of raw trace points. This is an improvement, but not a solution: 28 of 50 evaluated queries still have ratio above $2.0$, and the path oracle still reaches only $0.590$ path F1 and $0.261$ NDTW. The important result is therefore not that the proposed method is already superior to external systems, but that the project now has a concrete public-dataset evaluation path with uncertainty estimates and diagnostics.

\subsection{Result Summary}
The OSM public-trace experiment shows that the full pseudo-history pipeline is operational, but strict reconstruction quality is not yet sufficient. The Porto experiment provides a stronger public-dataset evaluation path: vehicle-aware profile ranking improves over shortest-distance and the generic profile on ranking metrics, while path metrics and oracle ceilings reveal that reconstruction and candidate generation are still the limiting factors. This combination is useful for a submission because it avoids hiding negative evidence while still showing a concrete technical contribution.
