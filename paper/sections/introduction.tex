Route recommendation is not only a shortest-path problem. A person may prefer a greener, quieter, simpler, better-lit, or more familiar route even when that route is not the minimum-distance option. This makes routing a natural personalization problem: the best route depends on the origin and destination, but also on the request context and the traveler's implicit or explicit preferences. Prior work on personalized route recommendation often assumes large-scale navigation logs, clean historical route choices, or trained models over dense trajectory datasets~\cite{dai2015bigtrajectory, nasr2019, userhabits2024}. Those assumptions are strong for a research prototype that aims to remain transparent, reproducible, and deployable with open geospatial data.

This paper studies a deliberately constrained alternative: can OSM-derived historical movement signals support preference-aware route ranking when clean user histories are unavailable? We use the term \emph{pseudo-history profile} to avoid overclaiming identity or intent. Public GPS traces can indicate where movement has occurred, but they are not necessarily persistent per-user logs. Treating them as clean preferences would make the evaluation look stronger than it is. Instead, the system explicitly measures reconstruction quality, marks weak reconstruction as a limitation, and evaluates route ranking separately from trace reconstruction.

We present a working OSM-based route-ranking system. Given an origin, destination, request context, optional preference text, and optional history records, the backend generates OSM route candidates, extracts interpretable route features, constructs dynamic profile weights from contextual history, and ranks candidates using profile, prompt, or hybrid modes. The implementation is exposed through a FastAPI backend and React interface, preserving the product-facing demonstration that motivated the project. This matters because route recommendation is ultimately an interactive decision aid: the system must produce ranked alternatives and explanations that a user can inspect, not only offline scores.

The paper also reframes the evaluation around validity. A first OSM public-trace experiment demonstrates the end-to-end pipeline for Toronto walking traces, but strict usability fails because route-distance ratio remains too high. We therefore add a public Porto taxi trajectory benchmark to move closer to recognized route-recovery evaluation. The Porto benchmark reconstructs observed OSM driving routes from public taxi trajectories, generates route candidates for the same origin and destination, and reports ranking metrics plus path-overlap metrics with bootstrap confidence intervals. This does not yet reproduce a full external learned baseline such as NASR~\cite{nasr2019}, but it makes the comparison target explicit and measurable.

The contributions are:

\begin{itemize}
  \item An end-to-end OSM route-ranking architecture that combines candidate generation, interpretable route features, contextual pseudo-history profiles, prompt-based ranking, and a GUI/API workflow.
  \item A trace-to-profile pipeline that converts public GPS-derived signals into pseudo-history records while reporting reconstruction quality instead of assuming clean user histories.
  \item A corrected route-candidate evaluation protocol in which generated candidates are ranked against held-out observed routes, avoiding a weaker history-record ranking setup that favored shortest-distance baselines.
  \item A Porto public-dataset benchmark with random, shortest-distance, profile, vehicle-profile, feature-oracle, and path-oracle comparisons using Hit@K, MRR, NDCG@3, feature distance, path F1, NDTW, edit distance, and confidence intervals.
\end{itemize}

The central claim is intentionally modest but useful: profile-aware OSM reranking can be evaluated transparently against non-personalized baselines, and early public-dataset results show promise, while reconstruction and candidate generation remain the main obstacles to a stronger conference claim.
