# Meeting Walkthrough

## 1. What Changed Since Last Meeting

The paper claim is now narrower and safer: this is an interpretable OSM route-candidate reranking framework, not a claim that we beat NASR or solve full personalized route generation.

We made Porto the main public benchmark path, added paired bootstrap comparisons, clarified the role of GPT/LLMs, and separated Toronto OSM traces from Porto taxi trajectories.

## 2. Main Research Question

Can trajectory-derived profiles rerank generated OSM route candidates closer to observed route behavior than non-personalized baselines under open-data constraints?

## 3. What Graph Is Used

The main graph is the OpenStreetMap road/path network. Nodes are intersections or map points, and edges are walkable or driveable road/path segments with attributes such as length, road type, lighting, crossings, tunnels, and geometry.

## 4. Background Pipeline

The system processes public GPS-derived traces, approximately map-matches them to OSM, extracts route features, builds contextual profiles, generates OSM route candidates for a new origin-destination query, and reranks those candidates.

## 5. Why Porto Was Added

Toronto OSM traces are useful for feasibility and pseudo-history diagnostics, but they are exploratory and walking-style. Porto is a recognized public taxi trajectory benchmark, so it is the stronger path for same-data evaluation and future external baseline reproduction.

## 6. Baselines

The current baselines are random, shortest-distance, generic profile, trajectory-derived vehicle profile, learned feature ranker, feature oracle, and path oracle. The oracle rows are diagnostic upper bounds, not deployable methods.

## 7. Metrics

Ranking metrics are Hit@1, Hit@3, MRR, NDCG@3, and mean feature distance. Path metrics are path precision/recall/F1, NDTW, edit distance, maximum nearest-point distance, route-distance ratio, and GPS-to-route distance. Porto also includes bootstrap confidence intervals and paired bootstrap differences.

## 8. Main Toronto OSM Trace Result

The Toronto probe shows the pipeline works end to end and produces usable exploratory pseudo-history signals. GPS-to-route proximity is strong, but route-distance ratio remains above the strict threshold, so it is not the main external benchmark.

## 9. Main Porto Result

In the 100-query Porto benchmark, the trajectory-derived vehicle profile improves ranking metrics over shortest-distance and the generic profile. Paired bootstrap differences are significant for Hit@3 and MRR versus shortest-distance, and significant across the main ranking metrics versus the generic profile. Path F1/NDTW still expose limitations, and shortest-distance remains competitive on path-level quality.

## 10. What We Are Not Claiming

We are not claiming to beat NASR. NASR learns neural costs inside A* search, while our system reranks candidates generated by OSM graph search. We are also not claiming clean human preference learning from public OSM traces or real prompt-based personalization on Porto.

## 11. Limitations

The main limitations are approximate map matching, candidate-pool ceiling, small Toronto held-out set, 100-query Porto scale, lack of persistent user identity in public traces, no direct NASR reproduction yet, and no real user-stated preference text in Porto.
